## Q1 — Scope decisions
**Compare your original BRIEF.md feature list to what you actually built. What did you cut, add, or change, and why?**

My original BRIEF.md listed these core features:
- User authentication (register and login)
- Create and manage personal habits
- Log daily completion of habits (once per day)
- Private dashboard displaying current streaks
- Anonymous public leaderboard showing top streaks

What I actually built matched the brief closely, but with some meaningful additions and changes:
- Changed: The "once per day" logging rule was replaced with a daily goal system — users set a goal (e.g. 8 glasses of water) and can log progress multiple times per day. The streak only increments when the daily goal is fully met. This made the app significantly more useful and realistic.
- Added: A stats bar on the dashboard showing total habits, today's completion count, and best streak.
- Added: Color-coded habit cards by category (Health = green, Fitness = blue, etc.) for better visual hierarchy.
- Added: Edit habit functionality - not in the original brief but felt essential once the app was running.
- Added: Tie handling on the leaderboard - users with the same streak share the same rank.
- Nothing was cut - all five original features were delivered.

The goal system was the biggest scope change, and it added complexity to both the back-end (removing the unique index on HabitLog, updating streak logic) and the front-end (progress counter, button states).

## Q2 — Technical challenge
**Describe the single hardest technical problem you encountered. What was the error or obstacle, and how did you resolve it?**

The hardest technical problem was the daily goal logging system breaking after I changed the streak logic.

Originally, HabitLog had a compound unique index { habitId, date } that prevented more than one log per habit per day. When I changed the system to allow multiple logs per day (up to the goal), I updated the code — but the old index was still sitting in MongoDB and kept rejecting the second log attempt with a 409 Conflict error.

The front-end showed the button as available but clicking it did nothing. The browser console showed a 409, which pointed to a duplicate key violation rather than a logic error.

The fix required dropping the index directly from MongoDB using a Node.js script:
```bash
node -e "
const mongoose = require('mongoose');
require('dotenv').config();
mongoose.connect(process.env.MONGODB_URI).then(async () => {
  await mongoose.connection.collection('habitlogs').dropIndex('habitId_1_date_1');
  console.log('Index dropped!');
  process.exit();
});"
```
This taught me that schema changes in code don't automatically update existing database indexes, you have to manage those separately. It also reinforced the importance of reading error status codes carefully: a 409 means conflict/duplicate, not a server crash.

## Q3 — AI and vibe-coding
**Reflect on your use of Claude Code during development. Give two specific examples: one where the AI output worked well with minimal changes, and one where you had to significantly debug or rewrite what it gave you. What did you learn from the difference?**

I used Claude throughout this project as my primary AI assistant.

Example where AI worked well with minimal changes:
The entire back-end scaffold — models, routes, middleware, and entry point — was generated by Claude in one session and worked almost immediately. I ran npm run dev, got Connected to MongoDB Atlas and Server running on port 3000 on the first try, and all Thunder Client tests passed. The streak calculation logic in habits.js was particularly well thought out — Claude normalized dates to midnight UTC and correctly handled the edge case of a broken streak resetting to 1. I didn't need to change any of it.

Example where I had to debug significantly:
The front-end CSS was more difficult. Claude generated a full stylesheet but it conflicted with Vite's default style.css — the old Vite styles were overriding my custom ones, making the login page unstyled for a long time. I had to manually identify the conflict, delete all the default Vite CSS, and rebuild the stylesheet from scratch. Claude hadn't accounted for the fact that Vite scaffolds with its own opinionated base styles.

What I learned:
AI is excellent at generating well-structured logic (back-end routes, data models, algorithms) but less reliable when the output needs to fit into an existing environment with its own configuration (like a Vite project with pre-existing styles). The more specific the environment, the more likely you are to need to debug.

## Q4 — Architecture
Explain how data flows through your application for one specific user action — from the moment the user clicks something, through the front-end, the API, the database, and back to the screen.

User action: clicking "Log it 🔥" on a habit card

1. Front-end (Dashboard.vue): The user clicks the button. Vue calls the ``markComplete(habitId)`` method, which makes a POST request using axios to ``${VITE_API_URL}/api/habits/${habitId}/log`` with the JWT token in the Authorization header.
2. Back-end (Express — habits.js): The request hits the ``POST /:id/log route``. The auth middleware runs first — it extracts the token from the header, verifies it using ``jwt.verify()``, and attaches ``userId`` to the request. The route handler then finds the habit in MongoDB, checks it belongs to the logged-in user, and normalizes today's date to midnight UTC.
3. Database (MongoDB Atlas): The route counts how many HabitLog documents exist for this habit today. If the count is less than the habit's goal, a new HabitLog document is created. If this log completes the goal, the Habit document is updated — ``currentStreak`` increments if the last completion was yesterday, or resets to 1 if a day was missed.
4. Response: The route returns a JSON object with the updated ``todayCount`` and ``goal``. If ``todayCount === goal``, the front-end shows a toast notification: "🎉 Streak updated! You crushed your goal today!" The dashboard then re-fetches all habit data, and the button updates to "🎉 Crushed it!" and becomes disabled.

## Q5 — If you had two more weeks
What would you add or improve first, and what technical approach would you use?

The first thing I would add is timezone-aware streak tracking. Currently, the app normalizes all dates to midnight UTC, which means the "day" resets at 1–2am for users in Prague depending on the season. A user who logs a habit at 1:30am thinks they're logging for today but the app counts it as tomorrow.

To fix this I would:
- Store the user's timezone during registration (using the browser's ``Intl.DateTimeFormat().resolvedOptions().timeZone``)
- Save it to the User model in MongoDB
- Pass it to all date calculations on the back-end using a library like ``date-fns-tz`` or ``dayjs`` with timezone support
- Normalize dates to midnight in the user's local timezone rather than UTC

The second thing I would add is habit history and a calendar view — a way to see which days you completed a habit over the past month. The data is already being stored in HabitLogs, so this would mainly be a front-end feature: a calendar component that colors days green or red based on whether logs exist for that date. This would make streaks feel much more meaningful and give users a visual sense of their progress over time.
